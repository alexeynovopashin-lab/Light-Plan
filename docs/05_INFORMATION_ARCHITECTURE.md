# Information Architecture

## Назначение документа

Этот документ описывает информационную модель продукта.
Он отвечает не на вопрос «какие будут экраны», а на вопрос:
Какие сущности существуют в мире Light Plan.
Экраны могут меняться.
Информационная модель должна оставаться стабильной.

## Главная сущность

Moment
Moment — центральная сущность приложения.
Фотограф никогда не работает с солнцем.
Он работает с конкретным моментом времени.
Каждый момент содержит описание состояния окружающего мира.
В него входят:
- время;
- положение солнца;
- положение луны;
- погодные условия;
- облачность;
- качество света;
- направление света;
- цветовая температура;
- длина теней;
- вероятность красивого света;
- рекомендации.
Практически всё приложение является разными представлениями Moment.

## Time

Время — главный навигационный слой продукта.
Не календарь.
Не карта.
Любая информация существует только относительно времени.
Все остальные сущности привязаны ко времени.

## Location

Локация описывает место съемки.
Она содержит:
координаты;
название;
часовой пояс;
ориентацию;
высоту;
избранное;
историю посещений;
заметки пользователя;
фотографии;
сценарии съемки.
Локация является постоянной сущностью.
Moment — временной.

## Light

Light — главная физическая сущность продукта.
Light объединяет множество параметров в одно описание.
Например:
Direction
Intensity
Softness
Warmth
Contrast
Shadow Length
Atmosphere
Visibility
Photographic Quality
В интерфейсе пользователь почти никогда не видит отдельные параметры.
Он видит состояние света.

## Weather

Погода является модификатором света.
Не самостоятельным объектом.
Погода существует только потому, что влияет на свет.

## Forecast

Forecast — последовательность будущих Moment.
Это не погодный прогноз.
Это прогноз изменений света.

## Session

Session — планируемая съемка.
Она объединяет:
локацию;
дату;
временной диапазон;
тип съемки;
участников;
оборудование;
заметки;
референсы.
В будущем Session становится центром профессионального рабочего процесса.

Session связана с Moment.
Поэтому приложение знает, попадает ли съемка в хороший свет, и замечает, если прогноз на нее испортился.
Это единственное, чего не умеет обычный календарь.

## Client

Клиент — постоянная сущность, а не строка внутри съемки.
Он содержит:
имя и контакты;
историю съемок;
даты, которые стоит помнить: день рождения, годовщину свадьбы;
суммы.

Клиент хранится на устройстве.
На нашем сервере не хранится ничего.
Резервная копия уходит в личное облако пользователя.

## Delivery

Delivery — обязательство, которое остается после съемки.
Оно содержит:
срок сдачи материала;
отметку «сдано»;
ссылку на готовые фотографии.

Срок берется из типа съемки: свадьба — три месяца, репортаж — три дня.
Либо задается вручную.
Либо отсутствует.

Съемка без Delivery закрыта.
Съемка с просроченным Delivery требует внимания.
Срочность — не состояние «горит или нет», а непрерывная величина.

## Place Memory

Каждая локация постепенно накапливает знания.
Когда лучше приезжать.
Какой свет бывает осенью.
Какие деревья закрывают солнце.
Когда появляются тени.
Лучшие точки съемки.
Это превращает приложение в личный атлас фотографа.

## Recommendation

Recommendation — результат анализа.
Это единственная сущность, которая непосредственно влияет на решение пользователя.
Например:
Go
Wait
Excellent
Risky
Too Cloudy
Leave in 18 minutes
Recommendation никогда не содержит технических терминов.

## Timeline

Timeline — способ просмотра изменений.
Практически любое состояние приложения можно представить через Timeline.
Сегодня.
Неделя.
Год.
День.
Золотой час.
Blue Hour.
Все это разные масштабы одной временной линии.

## Map

Карта не является главным экраном.
Она является способом выбора места.
Карта никогда не становится центром продукта.

## Notification

Уведомление не информирует.
Оно предлагает действие.
Например:
“Лучший свет начнется через 25 минут.”
а не
“Солнце достигло высоты 6°.”

## Camera Profile (будущее)

В перспективе приложение сможет учитывать особенности оборудования.
Размер сенсора.
Фокусное расстояние.
Динамический диапазон.
Тип объектива.
Но камера никогда не станет центральной сущностью.

## User

Пользователь хранит:
избранные места;
любимые часы;
предпочитаемые типы света;
историю съемок;
личные настройки.

## Отношения между сущностями

Location ↓ Moment ↓ Light ↓ Recommendation ↓ Decision
Session объединяет:
Location
Timeline
Forecast
Recommendation
User

## Информационная иерархия

В любой точке приложения информация располагается в следующем порядке.
- Решение
↓
- Свет
↓
- Время
↓
- Локация
↓
- Объяснение
↓
- Детали
↓
- Исходные данные
Этот порядок не должен нарушаться без серьезной причины.

## Главное правило архитектуры

В продукте нет экранов “про солнце”.
Нет экранов “про погоду”.
Нет экранов “про карту”.
Существуют только разные способы ответить фотографу на один вопрос:
“Когда и где я получу нужный свет?”
