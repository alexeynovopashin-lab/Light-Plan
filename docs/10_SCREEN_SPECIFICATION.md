# Screen Design Standard

## Назначение

Каждый экран Light Plan проектируется по одному шаблону.
Это гарантирует единый опыт использования независимо от количества функций.
Экран не начинается с компонентов.
Экран начинается с вопроса пользователя.

## Структура экрана

Каждый экран состоит из девяти обязательных разделов.

## 1. Purpose

Зачем существует экран.
Если Purpose невозможно сформулировать одним предложением — экран следует разделить.
Пример.
Today
Помочь пользователю решить, стоит ли проводить съемку сегодня.

## 2. User Question

Какой вопрос задает пользователь.
Примеры.
Стоит ли ехать?
Когда выйти?
Какой будет свет?
Как изменится ситуация через час?
Если экран отвечает более чем на один главный вопрос — это ошибка архитектуры.

## 3. Primary Decision

Главный вывод.
Он занимает наибольшее визуальное пространство.
Это всегда первое, что видит пользователь.
Примеры.
GO
WAIT
PERFECT LIGHT
BETTER TOMORROW
LEAVE IN 24 MIN

## 4. Supporting Evidence

Почему приложение пришло к такому выводу.
Не технические данные.
Именно причины.
Например.
Облака смягчат свет.
Ветер почти отсутствует.
Солнце пройдет между зданиями.
Контраст останется низким.

## 5. Context

Информация, которая помогает понять ситуацию.
Дата.
Время.
Локация.
Продолжительность окна.
Тип света.

## 6. Details

Самый нижний уровень.
Только здесь появляются:
азимут;
высота солнца;
UV;
давление;
астрономические параметры;
погодные модели.
Большинство пользователей никогда сюда не переходит.

## 7. Actions

Что пользователь может сделать дальше.
Количество основных действий:
не больше двух.
Идеально — одно.

## 8. Exit

После просмотра экрана должно существовать очевидное продолжение.
Например.
Открыть карту.
Посмотреть завтра.
Начать маршрут.
Создать съемку.

## 9. States

Каждый экран обязан быть описан во всех состояниях.
Loading
Offline
Empty
Success
Error
Unavailable
Forecast Updating
Location Denied
Все состояния проектируются одновременно.
Не после релиза.

## Информационная иерархия

Экран строится сверху вниз.
Решение
↓
Причина
↓
Контекст
↓
Действие
↓
Детали
Любое нарушение этой последовательности должно быть осознанным.

## Правило первого экрана

Первый экран приложения должен отвечать менее чем за пять секунд.
Если пользователю требуется анализировать интерфейс — экран не прошел проверку.

## Правило скролла

Самое важное никогда не должно требовать прокрутки.
Скролл существует только для изучения.
Не для принятия решения.

## Правило внимания

На экране всегда существует один визуальный центр.
Если взгляд пользователя начинает блуждать, композиция требует переработки.

## Правило воздуха

Свободное пространство является частью навигации.
Воздух направляет взгляд не хуже линий и стрелок.

## Правило цвета

Цвет никогда не используется для украшения.
Он сообщает состояние.
Если убрать цвет и экран перестанет быть понятным — архитектура построена неправильно.

## Правило текста

Текст не описывает интерфейс.
Он помогает принять решение.
Нельзя писать:
“Golden Hour Information”
Нужно писать:
“Best portrait light begins in 18 minutes.”

## Правило анимации

Любое движение сообщает изменение состояния.
Если убрать анимацию и смысл не изменится — анимация удаляется.

## Правило профессионала

Профессиональный фотограф должен понимать экран мгновенно.
Новичок должен понимать его благодаря естественному языку.
Мы никогда не создаем отдельные интерфейсы для разных уровней подготовки.

## Definition of Good Screen

Хороший экран отвечает пяти условиям.
После открытия пользователь понимает главное решение менее чем за пять секунд.
На экране существует только один главный объект внимания.
Информация раскрывается постепенно.
Нет декоративных элементов без функциональной роли.
После закрытия экрана пользователь уверен в следующем действии.
Если хотя бы один пункт нарушен — экран возвращается на этап проектирования.
